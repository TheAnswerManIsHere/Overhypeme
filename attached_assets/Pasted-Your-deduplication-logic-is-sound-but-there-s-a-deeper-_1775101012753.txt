Your deduplication logic is sound, but there’s a deeper issue worth reasoning through first: the Pexels search API is deterministic for a given (query, page) pair. If you’re hitting the same endpoint with “man doing pushups” and page=1 every time the user taps “random photo,” you’ll get the exact same result set every time. Three retries won’t help — you’ll get three identical duplicates and immediately conclude you’ve exhausted the library, even if there are 80 more photos on pages 2-8.
So the randomness has to come from somewhere. Two clean options:
Option A: Random page selection. First call with per_page=1 isn’t useful — instead, make an initial call to get total_results from the response metadata. Cache that count. Then on each “random photo” tap, pick a random page number from 1..total_results with per_page=1. This gives you true random sampling across the full result set. Your seen-URLs set then functions as a proper bloom filter against the full library, not just page 1.
Option B: Fetch a deep page upfront, shuffle client-side. Pull per_page=80 (Pexels max) on the first call, store all 80 IDs, shuffle them, and serve them sequentially. When the user exhausts the shuffled list, you know you’re truly done. This costs one API call instead of N, which matters for rate limits.
I’d lean toward Option B — it’s simpler, uses fewer API calls, and the exhaustion condition is trivially detectable (index >= array length) rather than heuristic.
Now, to your other question about failure modes beyond duplicates:
Rate limiting — Pexels returns 429 when you hit their rate cap. Surface this as “Image service is temporarily busy, try again later” and disable the button with a cooldown timer rather than letting the user hammer retries.
Zero results — Some search terms just return nothing (very niche or oddly-phrased queries). The API returns total_results: 0. Detect this on the initial fetch and never show the random photo button at all for that fact/gender variant. Fall back to a default gradient or abstract background.
Network/server errors (5xx, timeouts) — These are transient, not exhaustion. Don’t count them toward your “3 strikes” dedup logic. Retry with exponential backoff, and if it persists, show a generic “couldn’t load image” state with a manual retry option.
Dead image URLs — Pexels occasionally removes photos. The URL in your cache returns 404 when the browser tries to render it. Handle this with an onerror handler on the <img> tag that automatically advances to the next image in the pool and evicts the dead ID.
Invalid/expired API key — Returns 401. This is an ops problem, not a user-facing one. Log it aggressively, show the same graceful “no image available” fallback, and alert yourself.
So in summary: swap the retry-on-duplicate approach for a prefetch-and-shuffle model, and treat dedup exhaustion, zero results, and API errors as three distinct states with different UX responses rather than collapsing them all into one “no more photos” message.​​​​​​​​​​​​​​​​
