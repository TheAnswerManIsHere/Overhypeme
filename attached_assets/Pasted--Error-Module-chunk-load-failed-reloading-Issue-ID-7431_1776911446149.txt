# Error: Module chunk load failed — reloading

**Issue ID:** 7431369985
**Project:** sentry_project_frontend

## Issue Summary
Module Chunk Load Failure due to Potential Incomplete Build
**What's wrong:** Primary error is **Module chunk load failed** during dynamic import.
**In the trace:** Secondary errors show **SyntaxError** regarding missing bindings like 'getGetFactQueryKey'.

## Root Cause
```
# Root Cause of the Issue

Browser loads stale `index.html` referencing non-existent JavaScript chunks, causing dynamic import failures and subsequent `SyntaxError` for missing bindings.

### User navigates to application, browser loads cached index.html.
The browser, due to caching, loads an outdated `index.html` file. This file contains references to JavaScript chunk files from a previous deployment that no longer exist on the server.

### React app initializes, attempts to render root component.
The `main.tsx` entry point renders the `<App />` component, which sets up global providers and the main router.
(See @artifacts/overhype-me/src/main.tsx)

### Router component renders, triggering lazy page imports.
The `<Router />` component in `App.tsx` contains a `<Switch>` with multiple `Route` components, each dynamically importing a page using `React.lazy()`. For example, `FactDetail` is lazy loaded:
```typescript
const FactDetail = lazy(() => import("@/pages/FactDetail"));
```
When the user navigates to a route like `/facts/:id`, this dynamic import is triggered.
(See @artifacts/overhype-me/src/App.tsx)

### Browser requests non-existent JavaScript chunk for lazy page.
Because the `index.html` is stale, the browser requests a JavaScript chunk (e.g., `factdetail.a1b2c3d4.js`) that was part of the previous deployment but has been removed from the server in the current deployment. This results in a 404 HTTP error.

### Module chunk load fails, triggering internal error handler.
The browser's attempt to load the chunk fails. This failure is caught by an internal error handler (likely part of Vite's runtime or a custom `lazy-retry` mechanism, as indicated by the Sentry stack trace, even if the file isn't directly in the repo). This handler then throws a `Module chunk load failed` error.
```javascript
// Conceptual code from the 'lazy-retry.ts' mentioned in Sentry
// This is not directly in the codebase but inferred from the stack trace.
// It likely wraps the dynamic import to handle failures.
factory()
  .then(resolve)
  .catch(() => {
    window.location.reload();
    reject(new Error("Module chunk load failed — reloading")); // This is the error seen
  });
```

### Sentry ErrorBoundary catches the module load failure.
The `Sentry.ErrorBoundary` in `App.tsx` catches the `Module chunk load failed — reloading` error. This is the primary error reported to Sentry.
```typescript
function App() {
  return (
    <Sentry.ErrorBoundary fallback={({ resetError }) => <SentryFallback resetError={resetError} />}>
      {/* ... rest of the app ... */}
    </Sentry.ErrorBoundary>
  );
}
```
(See @artifacts/overhype-me/src/App.tsx)

### SentryFallback component triggers page reload.
The `SentryFallback` component, when rendered, provides a 'Reload page' button that explicitly calls `window.location.reload()`. The internal error handler also triggers a reload.
```typescript
// In SentryFallback.tsx
onClick={() => {
  resetError();
  window.location.reload();
}}
```
This reload is intended to fetch a fresh `index.html` and resolve the chunk loading issue.
(See @artifacts/overhype-me/src/components/SentryFallback.tsx)

### Subsequent errors: SyntaxError due to missing bindings.
If the initial module chunk load fails, any subsequent attempts to use exports from that module (e.g., `getGetFactQueryKey` from `@workspace/api-client-react` within `FactDetail`) will result in a `SyntaxError` because the module was never successfully loaded and parsed. This is a secondary symptom of the primary chunk load failure.
(See @lib/api-client-react/src/generated/api.ts)
```
