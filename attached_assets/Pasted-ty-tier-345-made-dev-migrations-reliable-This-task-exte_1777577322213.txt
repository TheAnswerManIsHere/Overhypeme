345 made dev migrations reliable. This task extends the same guarantee to production. You can’t ship one without the other, or you ship a system where dev works and prod silently drifts.
	•	Manual deploy steps are launch-day landmines. “Don’t forget to run migrations after deploy” is the kind of unwritten rule that survives exactly until the first time you forget. Solo dev + Replit + launch pressure = certain forgetting.
	•	The hard failure mode is the right design choice. Better for the server to refuse to start with a clear error than to start with a stale schema and serve broken responses. The task explicitly calls this out, which is good.
	•	Pre-launch is the right time. Deploy-path bugs are dramatically cheaper to fix when the only thing being deployed is your own development. Post-launch, every deploy-path issue happens with users on the system.
	•	The runner already exists per 345’s work, so this is mostly wiring + verification, not new infrastructure.
Modifications — verify a few production-specific risks Replit might miss:
	•	Confirm Replit’s deploy model. If Replit’s production runs as a single long-lived instance, this is straightforward — migrations run once at boot. If it ever scales to multiple instances or restarts frequently, multiple instances may try to run migrations concurrently on startup. The hash-based runner from 345 should be idempotent, but concurrent-safe is a stricter property. Ask Replit explicitly: “Confirm the migration runner is safe to invoke from multiple processes simultaneously, or add an advisory lock around it (pg_advisory_lock).”
	•	Startup latency impact. Long migrations delay the server accepting traffic, which can cause Replit’s healthcheck to fail and the deploy to abort. Worth knowing this exists; not worth pre-engineering for unless your migrations actually get long.
	•	Make sure failure logs go somewhere you’ll see them. A “deploy failed loudly with clear error” is only useful if you’ll actually see the error. Confirm it surfaces in whatever Replit deployment dashboard you watch — not just buried in container logs you’d have to dig for.
Concrete instruction to give Replit:
“Wire runMigrations() into the API server boot sequence so it runs before the server starts accepting traffic. If migrations fail, exit with non-zero status and a clear error message — do not start the server with a stale schema. Confirm the runner is safe to invoke concurrently if Replit ever runs multiple instances; add a pg_advisory_lock if not. Verify by deploying a small data-only migration and confirming it appears in the prod DB. Update .local/skills/deployment/SKILL.md if the deploy steps change.”
Watch-items:
	•	Don’t let Replit gate this behind a feature flag or env var. “Run migrations on startup unless SKIP_MIGRATIONS=true” sounds defensive but in practice means the first time something looks weird, someone will set the flag and forget. Migrations should always run on prod startup.
	•	Verify the deployment skill doc actually gets updated. It’s the kind of side-step Replit might skip, leaving you with documentation that contradicts behavior.
	•	Test the failure path before launch. Deploy a deliberately-broken migration once, confirm the server refuses to start and the error is loud and clear. Then revert. This proves the safety net actually catches you, rather than trusting it does.
