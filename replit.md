# Overview

Overhype.me is a community-driven, fully personalized facts database where facts are stored with dynamic tokens (e.g., `{First_Name}`) and rendered with user-defined names. The project aims to provide a platform for users to engage with facts, generate memes, and eventually include affiliate marketing and analytics.

The project is built as a pnpm workspace monorepo using TypeScript, leveraging Express 5 for the API, PostgreSQL with Drizzle ORM for data storage, and React with Vite for the frontend. Key capabilities include user authentication (Replit OAuth and email/password), AI-powered duplicate detection and moderation, meme generation with video capabilities, and comprehensive error reporting. Future plans include integrating affiliate links, Google Analytics/AdSense, and a "Fact of the Day" email service.

# System Architecture

The project is structured as a pnpm monorepo, separating the API server, frontend, database layer, and generated API clients into distinct packages.

**UI/UX Decisions:**
The frontend, `overhype-me`, is a React+Vite application featuring a dark theme with orange accents. It includes user-facing pages like a facts leaderboard, search, hashtag browsing, user profiles, and an admin interface. Meme generation is handled via an HTML5 Canvas builder.

**Technical Implementations:**
-   **Monorepo:** pnpm workspaces manage packages, each with its own dependencies.
-   **API Server (`api-server`):** Express 5 handles API routes, authentication, and interactions with the database. It uses Zod for request/response validation.
-   **Frontend (`overhype-me`):** React with Vite provides the user interface. It communicates with the API via generated React Query hooks.
-   **Database Layer (`db`):** PostgreSQL with Drizzle ORM manages data. The schema includes tables for users, facts, sessions, hashtags, ratings, comments, and activity feeds. `pgvector` is used for AI-driven duplicate detection.
-   **Authentication:** Supports both Replit OIDC and local email/password authentication using `bcryptjs` for password hashing.
-   **API Codegen:** OpenAPI 3.1 spec is used with Orval to generate TypeScript clients (`api-client-react`) and Zod schemas (`api-zod`), ensuring type safety and validation across the stack.
-   **AI Features:**
    -   **Duplicate Detection:** Utilizes `pgvector` for cosine similarity checks on 384-dimensional embeddings generated by OpenAI's `text-embedding-3-small` model.
    -   **Meme Video Generation:** Integrates with fal.ai's Kling image-to-video model to animate memes into short videos.
-   **Error Reporting:** Sentry is integrated for end-to-end error monitoring for both backend (Node/Express) and frontend (React), with PII scrubbing and source map uploads.
-   **Email Notifications:** Resend is used for transactional emails like verification, password resets, and review notifications.
-   **Build System:** `esbuild` is used for backend bundling, and `Vite` for frontend, with TypeScript compiler (`tsc`) handling type checking across the monorepo using composite projects and project references.

**Feature Specifications:**
-   **Fact Management:** Users can submit facts, which undergo AI duplicate detection. An admin review workflow is in place for flagged duplicates.
-   **Meme Generation:** Users can create memes with HTML5 Canvas, gradient themes, and upload them to cloud storage. Video generation is also supported.
-   **User Engagement:** Features include a facts leaderboard, search, hashtag browsing, ratings, comments, and a personalized activity feed.
-   **Admin Interface:** A dedicated `/admin` route tree provides a dashboard for managing facts (bulk import), users, billing, and review processes.

# Operations

## Dev workflow resilience
All three artifact dev workflows (`api-server`, `overhype-me`, `mockup-sandbox`) are wrapped in `scripts/dev-supervisor.sh`, which:
- Frees the bound port (`fuser -k $PORT/tcp`) before each (re)start, so a leftover process from the previous run no longer causes `EADDRINUSE`.
- Restarts the inner command on non-zero exit with backoff (1s â†’ 2s â†’ 3s â†’ 10s capped).
- Gives up after 20 crashes within 5 minutes so a hard-failing build doesn't bury the real error in scrolling logs â€” fix the underlying error and restart the workflow manually.
- Forwards `SIGTERM`/`SIGINT` to the child so workflow stop/restart behaves normally.

The api-server additionally has `process.on("uncaughtException")` and `process.on("unhandledRejection")` handlers that capture the error to Sentry, flush, and `process.exit(1)`. The supervisor then restarts the process automatically.
